==================================================================
  블로그 글 저장 방식 변화 — 파일(.md) → DB(H2) 하이브리드
  (글쓰기 기능 + DB 활용 설명 + 예상 Q&A)
==================================================================

------------------------------------------------------------------
[1] 무엇이 바뀌었나 (Before → After)
------------------------------------------------------------------

■ Before (기존 방식)
- 모든 글이 resources/posts/ 폴더의 .md(마크다운) 파일.
- 앱 시작 시 PostService가 .md를 전부 읽어 메모리에 보관.
- 글을 추가하려면 → 직접 .md 파일을 만들고 서버 재시작.
- 즉, "웹에서 글을 쓸 수 없음" (개발자가 파일로만 추가).

■ After (바뀐 방식 = 하이브리드)
- 기존 .md 글은 그대로 유지(읽기 전용 소스).
- 새로 쓰는 글은 웹 폼(/write)에서 작성 → H2 데이터베이스에 저장.
- 화면에는 ".md 글 + DB 글"을 합쳐서 최신순으로 보여줌.
- 즉, "웹에서 글쓰기 버튼으로 바로 글 발행" 가능.

■ 왜 하이브리드(A)로 했나
- 기존 27개 글(포트폴리오 내용)을 안 건드리고 보존.
- 동작하는 사이트를 깨지 않으면서 글쓰기 기능만 안전하게 추가.
- 나중에 원하면 .md 내용을 DB로 옮기고 .md를 제거하면 "DB only"가 됨.


------------------------------------------------------------------
[2] 왜 H2 + Spring Data JPA 인가
------------------------------------------------------------------
- H2 : 자바 안에서 도는 임베디드 DB. 별도 설치/서버 실행 불필요.
       "파일 모드"(jdbc:h2:file:./data/blogdb)라 껐다 켜도 데이터 유지.
       로컬 데모에 최적 (배포 없이 동작).
- Spring Data JPA : 인터페이스만 선언하면 save()/findAll()/findBySlug() 등을
       스프링이 자동 구현 → SQL을 거의 안 써도 됨. 초보 진입장벽 낮음.
- H2 콘솔(/h2-console) : 저장된 데이터를 브라우저에서 표로 확인 가능.


------------------------------------------------------------------
[3] DB를 어떻게 활용했나 (구성 요소)
------------------------------------------------------------------

(1) 설정 — application.yml
    spring.datasource.url: jdbc:h2:file:./data/blogdb   (파일에 저장)
    spring.jpa.hibernate.ddl-auto: update               (엔티티 보고 테이블 자동 생성)
    spring.h2.console.enabled: true                     (콘솔 켜기)

(2) 엔티티 — PostEntity.java  (DB 테이블과 짝이 되는 클래스 = "post" 테이블)
    @Entity, @Id @GeneratedValue(자동 증가 PK)
    필드: slug(고유), title, date, category, tags(콤마문자열), summary,
          contentMarkdown(@Lob, 본문 마크다운 원문)

(3) 리포지토리 — PostRepository.java  (DB 접근 통로)
    interface PostRepository extends JpaRepository<PostEntity, Long>
    - 기본 제공: save, findAll, findById, delete ...
    - 이름 규칙 자동 쿼리: findBySlug(slug), existsBySlug(slug)

(4) 서비스 — PostService.java  (.md + DB 합치는 핵심)
    - findAll(): .md 글(메모리) + DB 글(repository.findAll())을 합쳐 최신순 정렬
    - findBySlug(): .md 먼저 찾고, 없으면 DB에서 조회
    - toPost(PostEntity): DB 엔티티 → 화면용 Post(record)로 변환
                          (이때 마크다운을 commonmark로 HTML 변환)
    - createPost(...): 새 글을 PostEntity로 만들어 repository.save() → DB 저장
    - generateSlug(): 제목으로 URL용 slug 생성(중복이면 -2, -3 …)

(5) 컨트롤러 — BlogController.java
    GET  /write : 글쓰기 폼(write.html), 카테고리 목록 전달
    POST /write : @RequestParam으로 입력 받아 postService.createPost() →
                  작성된 글로 redirect:/post/{slug}

(6) 화면 — write.html
    제목/카테고리(드롭다운)/태그/요약/본문(마크다운) 입력 폼.

■ 핵심 설계 포인트
  - 화면용 Post(record)는 그대로 두고, DB용 PostEntity를 "따로" 만들었다.
  - 읽을 때 엔티티 → Post로 변환하므로 "기존 템플릿을 하나도 안 바꾸고"
    DB 글이 .md 글과 섞여서 나온다.


------------------------------------------------------------------
[4] 데이터 흐름 (글쓰기 한 번)
------------------------------------------------------------------
  사용자가 /write 폼 작성 후 [발행]
    → POST /write
    → BlogController.writeSubmit(@RequestParam 제목/카테고리/태그/요약/본문)
    → PostService.createPost() : slug 생성 → PostEntity 만들어 repository.save()
    → H2 DB의 post 테이블에 INSERT
    → redirect:/post/{slug}
    → 해당 글 페이지 표시 (DB에서 읽어 마크다운→HTML 변환)


==================================================================
  예상 질의응답 (Q&A)
==================================================================

[동작 원리]

Q1. 기존엔 글을 어떻게 저장했고, 왜 DB를 도입했나요?
A. 기존엔 .md 파일로만 글을 두고 서버가 시작 때 읽었습니다. 웹에서 글을 쓸 수
   없어서, 글쓰기 기능을 위해 DB(H2)를 도입했습니다.

Q2. '하이브리드'가 무슨 뜻인가요?
A. 기존 .md 글은 파일에서 그대로 읽고, 새 글만 DB에 저장한 뒤, 화면에선 둘을
   합쳐 보여주는 구조입니다. 기존 글을 안 건드리고 기능을 추가하려고 택했습니다.

Q3. .md 글과 DB 글이 어떻게 한 화면에 같이 나오나요?
A. PostService.findAll()이 메모리의 .md 글과 repository.findAll()의 DB 글을 합쳐
   날짜 최신순으로 정렬해 반환합니다. 템플릿은 그대로라 출처를 구분하지 않습니다.

[기술 선택]

Q4. 왜 MySQL이 아니라 H2인가요?
A. H2는 설치·서버 실행 없이 자바 안에서 도는 임베디드 DB라 로컬 데모에 적합합니다.
   파일 모드라 재시작해도 데이터가 유지되고, 콘솔로 확인도 됩니다. 배포 시
   MySQL로 바꿔도 JPA를 써서 코드는 거의 그대로입니다.

Q5. Spring Data JPA가 뭐고 왜 좋나요?
A. DB 접근을 자바 인터페이스로 다루게 해주는 기술입니다. JpaRepository를 상속한
   인터페이스만 선언하면 save/findAll/findBySlug 같은 메서드가 자동 생성돼,
   SQL을 직접 안 써도 됩니다.

Q6. findBySlug 같은 메서드는 직접 구현했나요?
A. 아니요. 메서드 이름 규칙(findBy + 필드명)만 선언하면 스프링이 쿼리를 자동
   생성합니다. existsBySlug도 마찬가지입니다.

[데이터 / 구조]

Q7. 'post' 테이블은 누가 만들었나요?
A. spring.jpa.hibernate.ddl-auto=update 설정이 PostEntity를 보고 테이블을
   자동으로 생성/갱신합니다. 직접 CREATE TABLE을 쓰지 않았습니다.

Q8. 엔티티(PostEntity)와 화면용 Post(record)를 왜 나눴나요?
A. DB 저장용과 화면 표시용 관심사가 다르기 때문입니다. 엔티티는 DB에, record는
   템플릿에 쓰고, 읽을 때 엔티티→record로 변환합니다. 덕분에 기존 화면 코드를
   안 바꿔도 됐습니다.

Q9. 본문(마크다운)은 어디서 HTML로 바뀌나요?
A. 저장은 마크다운 원문 그대로 DB에 하고, 화면에 보여줄 때 commonmark로 HTML로
   변환합니다(toPost 메서드). 원문을 보존해 나중에 재편집이 쉽습니다.

Q10. slug(URL 주소)는 어떻게 정해지나요?
A. 제목을 소문자로 바꾸고 특수문자를 -로 치환해 만듭니다. 같은 slug가 이미 있으면
    -2, -3을 붙여 중복을 피합니다.

[안정성 / 한계]

Q11. 서버를 껐다 켜도 글이 남나요?
A. 네. H2 파일 모드라 data/blogdb.mv.db 파일에 저장돼 재시작 후에도 유지됩니다.

Q12. 배포해야만 글쓰기가 되나요?
A. 아니요. 로컬(localhost:8080)에서 그대로 동작합니다. 배포는 공개 URL을 주는
    것일 뿐 기능과 무관합니다.

Q13. 현재 한계나 개선점은?
A. ① 카테고리 트리의 글 개수는 시작 시 1회 계산이라 새 글이 즉시 반영되지 않고
    재시작이 필요합니다. ② 글 수정/삭제 기능은 아직 없습니다. ③ 입력값 검증·
    인증(아무나 글쓰기 가능)이 없습니다. 추후 보완 가능합니다.

Q14. .md를 전부 지우면 어떻게 되나요?
A. 화면 결과는 DB 글만 남아 'DB only'와 같아집니다. 다만 .md를 지우면 그 글
    내용이 사라지므로, DB로 옮긴 뒤 지워야 합니다. 코드상 .md 읽는 부분은 남습니다.

 화면용 객체로 변환해 기존 화면 코드를 그대로 재사용했다."
