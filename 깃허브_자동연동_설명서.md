# 📘 블로그 ↔ GitHub 자동 연동 기능 — 완전 해설서

> 이 문서는 "내 블로그의 PROJECTS 페이지에 내 GitHub 공개 저장소가 자동으로 뜨는 기능"이
> **어떤 원리로, 어떤 기술로, 어떤 코드 흐름으로** 동작하는지 처음부터 끝까지 설명한다.
> 마지막의 **예상 질의응답(Q&A)** 까지 읽으면 이 기능을 남에게 막힘없이 설명할 수 있다.

---

## 1. 한 줄 정의

> **사용자가 `/projects` 페이지에 접속하면, 서버(Spring Boot)가 GitHub REST API를 호출해
> 내 공개 저장소 목록(JSON)을 받아오고, 그것을 자바 객체로 변환·가공한 뒤
> Thymeleaf 템플릿으로 HTML을 만들어 브라우저에 보여준다.**

핵심은 "**저장소 정보를 내가 직접 입력하는 게 아니라, GitHub에서 실시간으로 끌어온다**"는 것.
GitHub에 새 저장소를 만들면 코드 수정 없이 블로그에도 자동 반영된다.

---

## 2. 큰 그림 (전체 아키텍처 & 데이터 흐름)

```
[브라우저]                [내 Spring Boot 서버]                    [GitHub 서버]
   │                          │                                       │
   │  ① GET /projects         │                                       │
   ├─────────────────────────►│                                       │
   │                          │  ② BlogController.projects() 실행      │
   │                          │  ③ gitHubService.fetchRepos() 호출     │
   │                          │                                       │
   │                          │  ④ HTTP GET                           │
   │                          │     api.github.com/users/{id}/repos   │
   │                          ├──────────────────────────────────────►│
   │                          │                                       │
   │                          │  ⑤ JSON 배열 응답                      │
   │                          │◄──────────────────────────────────────┤
   │                          │  ⑥ Jackson이 JSON → GitHubRepo[] 변환  │
   │                          │  ⑦ fork 제외 + 최신순 정렬             │
   │                          │  ⑧ Model에 repos 담기                  │
   │                          │  ⑨ Thymeleaf가 projects.html 렌더링    │
   │  ⑩ 완성된 HTML            │                                       │
   │◄─────────────────────────┤                                       │
   │  (저장소 카드들이 보임)    │                                       │
```

**핵심 포인트**: 브라우저는 GitHub와 직접 통신하지 않는다. **내 서버가 중간에서 대신 호출**한다.
(이게 왜 중요한지는 6장 "설계 결정"과 Q&A에서 자세히.)

## 3. 사용 기술 스택과 각각의 역할

| 기술 | 역할 |
|---|---|
| **GitHub REST API** | 저장소 데이터의 출처(데이터 소스). HTTP로 요청하면 JSON으로 응답 |
| **Spring MVC (`@Controller`)** | URL 요청을 받아 처리할 메서드로 연결 |
| **Spring `RestClient`** | 내 서버가 GitHub로 HTTP 요청을 보내는 HTTP 클라이언트 (Spring 6.1+) |
| **Jackson** | GitHub가 준 JSON 문자열을 자바 객체(`GitHubRepo`)로 자동 변환(역직렬화) |
| **Java `record`** | 저장소 데이터를 담는 불변(immutable) 데이터 모델 |
| **Thymeleaf** | 자바 객체 목록을 HTML로 렌더링하는 서버 사이드 템플릿 엔진 |
| **`@Value` / application.properties** | 깃허브 아이디·토큰 같은 설정값을 코드 밖에서 주입 |

## 4. 코드 흐름 — 파일별 상세 해설

연동에 관여하는 파일은 4개 + 설정 1개다.

```
src/main/resources/application.properties        ← ① 설정값 (아이디/토큰)
src/main/java/.../service/GitHubService.java     ← ② API 호출의 핵심
src/main/java/.../model/GitHubRepo.java          ← ③ JSON을 담을 그릇
src/main/java/.../controller/BlogController.java ← ④ /projects 라우팅
src/main/resources/templates/projects.html       ← ⑤ 화면 렌더링
```

### ① application.properties — 설정값 분리
```properties
github.username=leesomin7747       # 내 깃허브 아이디
github.token=${GITHUB_TOKEN:}      # (선택) 토큰은 환경변수에서 주입, 없으면 빈 값
```
- 아이디/토큰을 **코드에 하드코딩하지 않고** 설정 파일/환경변수로 뺐다.
- `${GITHUB_TOKEN:}` 은 "환경변수 `GITHUB_TOKEN` 값을 읽되, 없으면 빈 문자열" 이라는 뜻.
- 토큰을 코드·깃에 노출하지 않기 위한 **보안 설계**.

### ② GitHubService.java — 실제로 GitHub를 호출하는 곳 (가장 중요)
```java
@Service
public class GitHubService {

    private final RestClient restClient;
    private final String username;

    // 생성자에서 설정값을 주입받아 RestClient를 1번만 구성
    public GitHubService(@Value("${github.username}") String username,
                         @Value("${github.token:}") String token) {
        this.username = username;

        RestClient.Builder builder = RestClient.builder()
                .baseUrl("https://api.github.com")                        // 공통 주소
                .defaultHeader("Accept", "application/vnd.github+json")    // JSON 형식 요청
                .defaultHeader("X-GitHub-Api-Version", "2022-11-28");      // API 버전 고정

        // 토큰이 있으면 인증 헤더 추가 (호출 한도 60 → 5000회/시간)
        if (token != null && !token.isBlank()) {
            builder.defaultHeader("Authorization", "Bearer " + token);
        }
        this.restClient = builder.build();
    }

    public List<GitHubRepo> fetchRepos() {
        try {
            GitHubRepo[] repos = restClient.get()
                    .uri("/users/{username}/repos?sort=updated&per_page=100&type=owner", username)
                    .retrieve()                       // 요청 전송
                    .body(GitHubRepo[].class);        // 응답 본문을 GitHubRepo 배열로 변환

            if (repos == null) return List.of();

            return java.util.Arrays.stream(repos)
                    .filter(r -> !r.fork())                                  // 포크 저장소 제외
                    .sorted(Comparator.comparing(GitHubRepo::updatedAt).reversed()) // 최신순
                    .toList();
        } catch (Exception e) {
            System.err.println("[GitHubService] 저장소 조회 실패: " + e.getMessage());
            return List.of();                         // 실패해도 빈 목록 → 페이지는 정상
        }
    }

    public String getUsername() { return username; }
}
```

**한 줄씩 의미**:
- `@Service` : 스프링이 이 클래스를 빈(bean)으로 관리 → 컨트롤러가 주입받아 사용.
- 생성자에서 `RestClient`를 **딱 한 번** 만들어 재사용(매 요청마다 새로 만들지 않음).
- `.baseUrl(...)` : 모든 요청의 공통 앞부분. 이후엔 `/users/...` 뒷부분만 지정.
- `Accept: application/vnd.github+json` : "JSON으로 응답해줘"라고 GitHub에 명시.
- `Authorization: Bearer <토큰>` : 토큰 인증. 없으면 익명 호출(한도 낮음).
- `.uri(".../repos?sort=updated&per_page=100&type=owner", username)` : 쿼리 파라미터
  - `sort=updated` 최근 수정순, `per_page=100` 한 번에 최대 100개, `type=owner` 내가 소유한 것만.
  - `{username}` 자리표시자에 변수 안전하게 치환.
- `.retrieve().body(GitHubRepo[].class)` : 요청 보내고, 응답 JSON을 **GitHubRepo 배열로 변환**.
- `.filter(!fork)` : 남의 저장소를 복제(fork)한 건 내 작업물이 아니므로 제외.
- `.sorted(updatedAt 역순)` : 최근 작업한 저장소가 위로.
- `try/catch` : 네트워크 오류·한도 초과 등 예외 발생 시 **빈 목록 반환** → 페이지가 깨지지 않음(우아한 실패).

### ③ GitHubRepo.java — JSON을 담을 그릇 (역직렬화 매핑)
```java
@JsonIgnoreProperties(ignoreUnknown = true)   // 모르는 필드는 무시
public record GitHubRepo(
        String name,
        String description,
        @JsonProperty("html_url") String htmlUrl,        // JSON의 html_url → htmlUrl
        String language,
        @JsonProperty("stargazers_count") int stars,     // JSON의 stargazers_count → stars
        @JsonProperty("forks_count") int forks,
        @JsonProperty("updated_at") String updatedAt,
        boolean fork
) {}
```
- GitHub가 주는 JSON 객체 하나 = `GitHubRepo` 하나.
- **`@JsonProperty`** : JSON 필드명(`html_url`, snake_case)과 자바 필드명(`htmlUrl`, camelCase)이
  다를 때 둘을 연결. 이게 없으면 매핑이 안 돼 값이 null이 된다.
- **`@JsonIgnoreProperties(ignoreUnknown = true)`** : GitHub 응답엔 필드가 수십 개인데
  내가 선언한 8개만 쓰고 **나머지는 무시**. 이게 없으면 모르는 필드 만났을 때 에러.
- `record` : getter·생성자·equals 등을 자동 생성하는 **불변 데이터 클래스**.

### ④ BlogController.java — /projects 요청 연결
```java
@GetMapping("/projects")
public String projects(Model model) {
    model.addAttribute("repos", gitHubService.fetchRepos());  // 저장소 목록을 화면에 전달
    model.addAttribute("pageTitle", "PROJECTS");
    return "projects";                                        // projects.html 렌더링
}
```
- `@GetMapping("/projects")` : `/projects` GET 요청을 이 메서드로 연결.
- `gitHubService.fetchRepos()` 결과를 `Model`에 `repos`라는 이름으로 담는다.
- `return "projects"` : `templates/projects.html`을 렌더링하라는 뜻(뷰 이름).
- 참고로 `commonAttributes()`(@ModelAttribute)가 모든 페이지에 `githubUsername`을 자동 추가.

### ⑤ projects.html — 받은 데이터를 화면으로
```html
<a class="repo-card" th:each="repo : ${repos}"          <!-- repos를 하나씩 반복 -->
   th:href="${repo.htmlUrl}" target="_blank">
    <h3 th:text="${repo.name}">repo-name</h3>
    <p th:text="${repo.description != null ? repo.description : '설명 없음'}">설명</p>
    <div class="repo-meta">
        <span th:if="${repo.language != null}" th:text="${repo.language}">Java</span>
        <span>★ <span th:text="${repo.stars}">0</span></span>
    </div>
</a>
<p th:if="${repos.isEmpty()}">저장소를 불러오지 못했습니다…</p>
```
- `th:each="repo : ${repos}"` : 컨트롤러가 넘긴 `repos` 리스트를 반복 → 저장소마다 카드 1개.
- `th:text` 로 이름·설명·언어·스타 수를 채우고, `th:href`로 저장소 링크 연결.
- `th:if="${repos.isEmpty()}"` : 비었으면(실패 등) 안내 문구.

## 5. 핵심 개념 깊이 이해하기 (원리)

### (1) REST API와 HTTP
- **API**: 프로그램끼리 약속된 방식으로 데이터를 주고받는 창구.
- **REST API**: HTTP 메서드(GET/POST…)와 URL로 자원을 다루는 방식.
  여기선 **GET**으로 "이 사용자의 저장소들을 조회"한다.
- 요청 `GET https://api.github.com/users/leesomin7747/repos` →
  응답: 저장소 정보가 담긴 **JSON 배열** `[ {...}, {...} ]`.

### (2) JSON 역직렬화 (Deserialization)
- GitHub가 주는 건 그냥 **문자열(JSON)** 이다.
- 자바에서 다루려면 객체로 바꿔야 함 → 이 과정이 **역직렬화**.
- Spring이 내부적으로 **Jackson**(HttpMessageConverter)을 써서
  `body(GitHubRepo[].class)` 한 줄로 JSON → 자바 객체 배열로 변환해준다.
- 필드명이 다르면 `@JsonProperty`로 다리를 놓아준다.

### (3) 왜 서버가 호출하나? (서버 사이드 fetch)
브라우저(자바스크립트)에서 직접 GitHub를 부를 수도 있는데, 굳이 서버가 부른다. 이유:
1. **토큰 보안**: 토큰을 브라우저 JS에 넣으면 누구나 본다. 서버에 두면 안 보인다.
2. **CORS 회피**: 서버 간 통신은 브라우저의 동일출처정책(CORS) 제약이 없다.
3. **SSR(서버 사이드 렌더링)**: 완성된 HTML을 내려주므로 검색엔진 노출·초기 로딩에 유리.
4. **가공·캐싱을 서버에서 통제** 가능(정렬·필터·향후 캐싱).

### (4) 인증과 호출 한도(Rate Limit)
- 토큰 **없이**: IP당 **시간당 60회**.
- 토큰 **있으면**: **시간당 5,000회**.
- 토큰은 `Authorization: Bearer <토큰>` 헤더로 전달. (개발/시연엔 토큰 없어도 충분)

### (5) 우아한 실패(Graceful Degradation)
- `fetchRepos()`는 예외가 나도 **빈 리스트**를 반환한다.
- 그래서 GitHub가 죽거나 인터넷이 끊겨도 **PROJECTS 페이지 자체는 정상 출력**(저장소만 안 보임).
- "한 기능의 실패가 전체 페이지를 깨뜨리지 않게" 하는 안정성 설계.

## 6. 설계 결정 & 트레이드오프 (왜 이렇게 짰는가)

| 결정 | 이유 | 대안 / 트레이드오프 |
|---|---|---|
| `RestClient` 사용 | Spring 6.1+ 표준, 동기식이라 코드가 단순·직관적 | `RestTemplate`(구식), `WebClient`(비동기, 여기선 과함) |
| 서버에서 호출 | 토큰 보안 · CORS 회피 · SSR | 클라이언트 fetch(토큰 노출·CORS 문제) |
| `record` 모델 | 불변·간결, DTO에 적합 | 일반 class(보일러플레이트 많음) |
| `@JsonIgnoreProperties` | API가 필드 추가해도 안 깨짐 | 없으면 응답 변화에 취약 |
| 실패 시 빈 목록 | 페이지 안정성 | 예외 전파(사용자에 500 에러) |
| 매 요청 호출(캐싱 없음) | 구현 단순, 항상 최신 | 트래픽·한도 부담 → **캐싱이 개선 1순위** |

## 7. 현재 한계와 개선 방향 (발표 때 "더 발전시킨다면?" 대비)

1. **캐싱 없음** → 페이지 열 때마다 GitHub 호출. 
   개선: `@Cacheable`(Spring Cache)로 N분 캐싱 → 한도 절약·속도 향상.
2. **저장소 100개 초과 시 잘림** (`per_page=100`이 1페이지 최대).
   개선: 페이지네이션 루프(`page=1,2,…`)로 전체 수집.
3. **동기 호출** → 응답 느리면 페이지도 느려짐.
   개선: 비동기(`WebClient`) 또는 백그라운드 갱신 + 캐시 서빙.
4. **언어/스타 외 정보 부족** → 토픽·핀 고정 저장소·README 미리보기 추가 가능.
5. **실패가 조용함** → 로깅을 `System.err` 대신 정식 로거(SLF4J)로, 모니터링 연계.

## 8. 🙋 예상 질의응답 (기술 면접/발표 대비)

### A. 동작 원리
**Q1. 이 기능 한 문장으로 설명해보세요.**
A. 사용자가 /projects에 접속하면 서버가 GitHub REST API를 호출해 공개 저장소 JSON을 받고,
Jackson으로 자바 객체로 변환·정렬한 뒤 Thymeleaf로 HTML을 만들어 응답합니다.
저장소를 직접 입력하지 않고 실시간으로 끌어오는 게 핵심입니다.

**Q2. 브라우저가 GitHub를 직접 부르지 않고 왜 서버가 부르나요?**
A. 세 가지입니다. ① 토큰을 브라우저에 노출하지 않으려고(보안), ② 서버 간 통신이라 CORS 제약이
없고, ③ 완성된 HTML을 내려주는 SSR이라 SEO·초기 로딩에 유리합니다.

**Q3. 새 저장소를 만들면 코드 수정 없이 어떻게 자동 반영되나요?**
A. 저장소 목록을 하드코딩이 아니라 매 요청마다 GitHub API에서 가져오기 때문입니다.
GitHub 상태가 곧 화면이므로, 새로 만들면 다음 접속 때 자동으로 나타납니다.

### B. 네트워크 / API
**Q4. 어떤 HTTP 요청을 보내나요?**
A. `GET https://api.github.com/users/{아이디}/repos?sort=updated&per_page=100&type=owner` 입니다.
Accept 헤더로 JSON을, X-GitHub-Api-Version으로 API 버전을 명시합니다.

**Q5. `sort=updated`, `type=owner`, `per_page=100`은 각각 뭔가요?**
A. 최근 수정순 정렬, 내가 소유한 저장소만(포크/협업 제외 경향), 한 페이지에 최대 100개입니다.

**Q6. 인증은 어떻게 하나요? 토큰이 꼭 필요한가요?**
A. 선택입니다. 토큰 없이도 시간당 60회 호출됩니다. 토큰을 `Authorization: Bearer` 헤더에 넣으면
5,000회로 늘어납니다. 토큰은 환경변수로 주입해 코드/깃에 노출되지 않게 했습니다.

**Q7. 호출 한도(rate limit)를 초과하면요?**
A. GitHub가 403을 반환합니다. 현재는 예외 처리로 빈 목록을 반환해 페이지는 정상 동작하고,
개선책으로 캐싱을 두면 호출 횟수를 크게 줄일 수 있습니다.

### C. JSON 변환 / Jackson
**Q8. JSON이 어떻게 자바 객체가 되나요?**
A. Spring이 내부적으로 Jackson(HttpMessageConverter)을 사용합니다. `body(GitHubRepo[].class)`
한 줄로 응답 JSON 배열을 GitHubRepo 배열로 역직렬화합니다.

**Q9. `@JsonProperty`는 왜 붙였나요?**
A. JSON 필드는 `html_url`처럼 snake_case인데 자바는 `htmlUrl` camelCase를 쓰기 때문입니다.
이름이 다르면 매핑이 안 돼 null이 되므로 `@JsonProperty("html_url")`로 연결해줍니다.

**Q10. `@JsonIgnoreProperties(ignoreUnknown=true)`는요?**
A. GitHub 응답엔 필드가 수십 개인데 저는 8개만 씁니다. 이 어노테이션이 없으면 선언 안 한
필드를 만났을 때 역직렬화가 실패합니다. 이걸로 모르는 필드는 무시해 API 변화에 견고해집니다.

**Q11. 왜 class가 아니라 record를 썼나요?**
A. 단순히 데이터를 담아 화면에 보여주는 DTO라 불변이면 충분하고, record는 생성자·getter·
equals/hashCode를 자동 생성해 코드가 간결해집니다.

### D. 스프링 / 구조
**Q12. RestClient는 왜 생성자에서 한 번만 만드나요?**
A. RestClient는 재사용 가능한 객체라 요청마다 새로 만들 필요가 없습니다. 빈 생성 시 한 번
구성해두면 커넥션 설정·헤더를 재사용해 효율적입니다.

**Q13. `@Service`, `@Controller`, `@Value`의 역할은?**
A. `@Service`는 비즈니스 로직 빈(여기선 GitHub 호출), `@Controller`는 웹 요청 처리,
`@Value`는 application.properties 값을 필드/생성자에 주입합니다.

**Q14. RestClient vs RestTemplate vs WebClient, 왜 RestClient?**
A. RestTemplate은 구식(유지보수 모드), WebClient는 비동기라 이 단순 동기 작업엔 과합니다.
RestClient는 Spring 6.1+의 현대적 동기 클라이언트라 가독성과 적합성이 가장 좋습니다.

**Q15. Model에 담은 repos가 어떻게 화면에 나오나요?**
A. 컨트롤러가 `model.addAttribute("repos", ...)`로 담으면, Thymeleaf가 `th:each="repo:${repos}"`
로 반복하며 저장소마다 카드 HTML을 생성합니다.

### E. 안정성 / 보안 / 성능
**Q16. GitHub가 응답하지 않으면 페이지가 깨지나요?**
A. 아니요. `fetchRepos()`가 try/catch로 예외를 잡아 빈 목록을 반환하므로 페이지는 정상 뜨고
저장소 영역만 안내 문구가 표시됩니다. (우아한 실패)

**Q17. 토큰을 어떻게 안전하게 관리하나요?**
A. 코드에 직접 쓰지 않고 `github.token=${GITHUB_TOKEN:}`로 환경변수에서 읽습니다. 토큰 문자열은
.gitignore된 파일이나 셸 환경변수에만 두어 깃 저장소에 올라가지 않습니다.

**Q18. 매번 호출하면 느리거나 한도에 걸리지 않나요? 어떻게 개선하죠?**
A. 맞습니다, 현재 한계입니다. Spring Cache(`@Cacheable`)로 결과를 수 분간 캐싱하면 호출 횟수를
줄이고 응답도 빨라집니다. 더 나아가 백그라운드 주기 갱신 + 캐시 서빙도 가능합니다.

**Q19. 저장소가 100개가 넘으면요?**
A. 현재는 per_page=100이라 100개까지만 가져옵니다. 개선하려면 `page` 파라미터를 1씩 늘리며
빈 응답이 올 때까지 반복 호출해 전체를 모으는 페이지네이션이 필요합니다.

**Q20. 이 기능을 더 발전시킨다면?**
A. ① 결과 캐싱, ② 페이지네이션, ③ 핀 고정/토픽/README 미리보기 추가, ④ 커밋 잔디·언어 통계
시각화(차트), ⑤ 비동기 호출로 응답 개선 등을 생각하고 있습니다.

## 9. 한눈에 보는 핵심 용어
- **REST API**: HTTP로 자원을 주고받는 약속. GET으로 조회.
- **JSON**: 데이터 표현 문자열 포맷. API 응답 형식.
- **역직렬화(Deserialization)**: JSON 문자열 → 자바 객체로 변환. (Jackson 담당)
- **RestClient**: 내 서버가 외부 API로 HTTP 요청을 보내는 도구.
- **SSR(서버 사이드 렌더링)**: 서버가 완성된 HTML을 만들어 응답.
- **Rate Limit**: 시간당 API 호출 허용 횟수(60/5000).
- **Graceful Degradation**: 일부 기능이 실패해도 전체는 동작하게 하는 설계.
- **DTO**: 데이터 전달용 객체(여기선 `GitHubRepo`).
